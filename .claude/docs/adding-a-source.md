# Как добавить источник вакансий

Источник — реализация `core.interfaces.JobSource`. Ядро не знает о его внутренностях.

## Шаги

1. **Папка.** Создай `sources/<name>/` с `__init__.py`, `source.py` (и `mapping.py`, `client.py` при нужде).
2. **Класс источника.** Реализуй `JobSource`:
   - `name: str` — уникальное имя (совпадает с ключом в `config.yaml`).
   - `capabilities` → `Capabilities(can_scan, can_search, can_apply)`. **Честно** выставь флаги.
   - `connect()` / `close()`.
   - `iter_dialogs(include_archived)` → `core.models.Dialog`.
   - `iter_new_messages(dialog, cursor)` → `core.models.RawMessage`.
   - опц. `search(query)`.
3. **Маппинг.** Преобразуй сырые объекты провайдера в модели `core/`. Телеграм/LinkedIn-специфика
   не должна протекать в ядро.
4. **Регистрация.** В `registry.py` добавь ветку под `config.sources.<name>.enabled`.
5. **Конфиг.** Добавь секцию `sources.<name>` в `config.yaml` (по умолчанию `enabled: false`).
6. **Курсор.** Реши, что такое cursor для источника (id/дата). Ядро его не интерпретирует.
7. **Тесты.** Маппинг и предфильтр — с моками.
8. **Документация.** `CLAUDE.md` в папке источника + строка в `sources/CLAUDE.md`.

## Capability-флаги

| Флаг | Смысл |
|---|---|
| `can_scan` | Перебор диалогов + чтение новых сообщений. |
| `can_search` | Серверный поиск по запросу. |
| `can_apply` | Откликаться. **У нас всегда False** — отклик только вручную. |

## ⚠️ Правовое / ToS

- **Telegram:** только чтение под аккаунтом пользователя (read-only), мягкие задержки, FloodWait.
- **LinkedIn:** автоматизированный сбор ограничен ToS. Предпочитать официальный API; свериться с условиями.
- **Веб-доски:** у каждой свои условия и rate limits. Предпочитать официальные API/RSS, не скрейпинг.
- Часть источников read-only по своей природе — отражай это во флагах и не реализуй отправку.

> Модель угроз и ограничения — `security.md`.
